# AGENTS.md — 全局开发指南（通用原则）

## 0. 适用范围

本文件是全局默认规则，适用于所有项目。项目根目录或子目录存在自己的 AGENTS.md 时，以更具体
的规则为准（就近覆盖），未被覆盖的部分仍然生效。

技术栈相关的具体规范（后端框架、前端框架、数据库、测试与启动命令等）请在各项目自己的
AGENTS.md 中补充，全局文件只约定跨项目通用的原则。

---

## 1. 核心原则

- 高效：优先解决实际问题，避免过度流程化。
- 稳定：默认最小变更，不破坏现有功能。
- 准确：先理解上下文，再修改代码；结论必须有依据，不臆测。
- 可维护：结构清晰、职责分离、可测试、可回滚。
- 中文化：沟通说明、注释、文档、提交信息默认使用中文；**代码标识符（变量名、函数名、类名）
  使用英文**。

---

## 2. 工作方式

1. **理解需求**：先读相关代码、配置、报错和上下文，不臆测。**若项目根目录下存在 `.ai-organization/`，优先加载其中的组织政策、部门岗位定义与 Skills。**
2. **明确范围**：只改和当前任务直接相关的内容，不做无关重构，不主动扩大需求范围。
3. **制定计划**：复杂任务先列简短步骤，简单任务直接处理；避免一次性大范围改动。**在组织化项目中，必须依照 `templates/project-raci.md` 定义分工，并通过 `skills/organization-manager/SKILL.md` 校验能力缺口，严禁单 Agent 越权决策或独立执行复杂业务闭环。**
4. **验证结果**：优先运行与改动相关的构建、测试或静态检查；无法运行需说明原因。
5. **总结交付**：按第 9 节模板总结，任务规模越大总结越完整，小任务可简化。

---

## 3. 安全约束与执行权限

**必须遵守：**

- 禁止泄露密钥、Token、密码、Cookie、内网地址等敏感信息，包括日志、注释、提交说明中。
- 禁止执行破坏性命令，例如 `rm -rf`、强制清库、覆盖生产数据、对共享分支 `force-push`。
- 禁止主动触发生产发布、生产部署、线上数据迁移；可以阅读、分析、修改 CI/CD 配置，但不得
  主动执行生产流水线。
- 数据库结构变更（删字段、改类型、改枚举值等）必须先说明影响范围和回滚方案，经用户确认后
  再执行。
- 破坏性操作、依赖大版本升级、跨模块重构，必须经用户明确授权后才能执行——仅提出方案不算授权。

**命令执行原则：**

- 执行命令前先查看项目实际脚本，不要凭经验猜测命令名。
- 只读、无副作用的检查类命令（lint、type-check、单元测试、本地构建）可以直接执行。
- 有副作用或不可逆的操作（联网请求、安装新依赖、提交合并、执行迁移脚本）先说明意图，等待
  确认后再执行。

---

## 4. 变更策略

- 优先最小变更，优先修复根因而非掩盖报错，优先复用现有架构、工具和代码风格。
- 明显无用、错误或过时的代码可以删除，但删除前必须确认未被引用。
- 不做占位实现：禁止遗留 `TODO`、`NotImplemented`、伪代码式交付。
- 只有用户明确要求重构、替换旧方案或清理历史包袱时，才允许破坏性改动，并需说明影响范围、
  迁移方式、回滚方案、需要人工确认的事项。

---

## 5. 依赖与文档

- 新增依赖前确认：项目中是否已有等价能力、是否主流稳定且维护活跃、是否有明显安全风险、
  是否兼容当前框架版本。不得擅自升级依赖大版本，确需升级需说明原因、影响范围和回滚方案。
- 需要补充注释或文档的场景：复杂业务规则、状态流转、权限判断、事务处理、数据迁移、定时
  任务、第三方接口对接、非显而易见的兼容处理。注释要解释"为什么这样做"，不复述代码表面
  含义。文档默认中文，文件编码统一 UTF-8。

---

## 6. Git 与提交

提交信息使用 Conventional Commits，例如：

```
feat: 新增用户管理列表筛选功能
fix: 修复登录失效后未跳转登录页的问题
refactor: 重构订单状态流转逻辑
```

每次提交对应一个目的明确的小变更，确认无误后再提交，不打包无关改动。可以用 `gh` 进行
PR / issue 相关操作，但不主动执行 `merge`、`force-push` 等有风险操作，除非用户明确要求。

---

## 7. 外部资料与工具使用

优先使用项目内代码、配置、文档作为依据。需要查询外部资料时：优先官方文档 and 权威来源，
说明来源和时效性，不上传项目敏感代码、密钥或业务数据。指定工具不可用时，说明原因并使用
可行的替代方式完成任务，不要静默跳过。

---

## 8. 禁止行为

- 未理解上下文就大范围改代码；为消除报错而删除核心逻辑；用占位代码假装完成任务。
- 随意吞掉异常、随意使用 `any` / 弱类型、硬编码魔法值。
- 未说明影响就修改数据库结构。
- 未经确认擅自升级依赖大版本、执行生产发布或破坏性命令。
- 泄露敏感信息；输出无法落地的空泛方案。

---

## 9. 默认交付模板

小任务按第 2 节简化说明即可；中大型改动（新增接口、修改业务流程、模块重构）使用以下结构：

```
## 结论
一句话说清：做了什么、现在什么状态。
✅ 完成 / ⚠️ 部分完成，待确认 / ❌ 被阻塞，需要你决定

## 变更内容
- 改了什么、为什么这么改（讲清楚，不写"修改点 1"这种占位）

## 验证情况
- ✅ 已执行 xxx，结果：通过
- ⚠️ 未执行 xxx，原因：...

## 风险与需要你确认的事
- 影响范围 / 兼容性 / 数据风险（没有就写"无明显风险"，不留空）
- 需要人工处理的事项（没有就省略这行）

## 后续建议（可选，没有就整节省略）
- ...

---
用到搜索或外部工具时才附加：用了什么工具、查了什么资料、改了哪些文件。
```

## 十、AI 组织化协同规范（若项目包含 .ai-organization）

当且仅当项目根目录存在 `.ai-organization/` 目录时，必须强制激活以下规则：

1. **角色与身份契合**：Agent 不再是独立的代码工具，而是拥有特定职责的“组织成员”。开发时必须根据 `registry/agent-registry.yaml` 明确自身岗位 ID，并在 `templates/agent-job-description.md` 定义的授权边界内活动。
2. **能力缺口与招聘**：遇到当前活跃 Agent 均不持有的能力（例如：算薪、法务合规）时，禁止“假装”拥有该技能。必须通过 AI HR 经理生成 `templates/job-request.md` 申请编制并运行招聘及面试流程。
3. **人机升级协作 (Escalation)**：在涉及财务预算、法务合规、敏感企业制度变更或无法调和的死锁冲突时，必须挂起任务并生成 `human-decision-required.md`，升级至 L4 级（人类专家）进行终审签字。
4. **决策投票与会签**：重大架构调整或数据库 Schema 变更，必须按照 `core/decision-policy.md` 执行多部门会签投票，并在 `templates/decision-record.md` 中留下共识归档。
